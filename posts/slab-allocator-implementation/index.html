Implementing Custom Slab Allocator with C++

implementing a slab allocator: high-performance memory management Link to heading

Slab allocators are one of the most efficient memory allocation strategies for high-frequency allocations of fixed-size objects. Originally developed by Sun Microsystems for the Solaris kernel, slab allocators have found their way into many high-performance systems.

What is a Slab Allocator? Link to heading

A slab allocator pre-allocates memory in fixed-size chunks (slabs) and maintains free lists for each size class. When you request memory, it simply pops an object from the appropriate free list - an O(1) operation.

Core Concepts Link to heading

1. Slab Link to heading

A contiguous chunk of memory divided into equal-sized objects.

2. Free List Link to heading

A linked list of available objects within a slab.

3. Size Classes Link to heading

Predefined object sizes that the allocator supports.

Basic Implementation Link to heading

Here’s a simplified slab allocator implementation in C++:

#include <vector>
#include <list>
#include <cstddef>

class SlabAllocator {
private:
    struct Slab {
        void* memory;
        size_t objectSize;
        size_t objectCount;
        std::list<void*> freeList;
        bool isFull;
    };

    std::vector<Slab> slabs_;
    size_t slabSize_;
    size_t objectSize_;

public:
    SlabAllocator(size_t objectSize, size_t objectsPerSlab = 1000)
        : objectSize_(objectSize), slabSize_(objectSize * objectsPerSlab) {
        // Align object size to cache line
        objectSize_ = (objectSize_ + 63) & ~63;
    }

    void* allocate() {
        // Find a slab with free objects
        for (auto& slab : slabs_) {
            if (!slab.freeList.empty()) {
                void* obj = slab.freeList.front();
                slab.freeList.pop_front();
                return obj;
            }
        }

        // Create new slab if needed
        createNewSlab();
        return allocate();
    }

    void deallocate(void* ptr) {
        // Find the slab containing this pointer
        for (auto& slab : slabs_) {
            if (ptr >= slab.memory && 
                ptr < static_cast<char*>(slab.memory) + slabSize_) {
                slab.freeList.push_back(ptr);
                return;
            }
        }
    }

private:
    void createNewSlab() {
        Slab slab;
        slab.memory = ::operator new(slabSize_);
        slab.objectSize = objectSize_;
        slab.objectCount = slabSize_ / objectSize_;
        slab.isFull = false;

        // Initialize free list
        char* mem = static_cast<char*>(slab.memory);
        for (size_t i = 0; i < slab.objectCount; ++i) {
            slab.freeList.push_back(mem + i * objectSize_);
        }

        slabs_.push_back(slab);
    }
};

Advanced Features Link to heading

1. Multiple Size Classes Link to heading 

class MultiSlabAllocator {
private:
    std::vector<SlabAllocator> allocators_;
    std::vector<size_t> sizeClasses_;

public:
    MultiSlabAllocator() {
        // Common size classes: 8, 16, 32, 64, 128, 256, 512, 1024
        sizeClasses_ = {8, 16, 32, 64, 128, 256, 512, 1024};
        for (size_t size : sizeClasses_) {
            allocators_.emplace_back(size);
        }
    }

    void* allocate(size_t size) {
        // Find appropriate size class
        for (size_t i = 0; i < sizeClasses_.size(); ++i) {
            if (size <= sizeClasses_[i]) {
                return allocators_[i].allocate();
            }
        }
        // Fall back to system allocator for large objects
        return ::operator new(size);
    }
};

2. Thread Safety Link to heading 

#include <mutex>

class ThreadSafeSlabAllocator {
private:
    SlabAllocator allocator_;
    mutable std::mutex mutex_;

public:
    void* allocate() {
        std::lock_guard<std::mutex> lock(mutex_);
        return allocator_.allocate();
    }

    void deallocate(void* ptr) {
        std::lock_guard<std::mutex> lock(mutex_);
        allocator_.deallocate(ptr);
    }
};

Performance Characteristics Link to heading

Advantages: Link to heading

  • O(1) allocation/deallocation for common cases
  • Excellent cache locality due to object clustering
  • Reduced fragmentation through size classes
  • Predictable performance with minimal variance

Disadvantages: Link to heading

  • Memory overhead for size class management
  • Fixed object sizes limit flexibility
  • Complex implementation for advanced features

Real-World Usage Link to heading

Slab allocators are used in:

  • Linux kernel (kmem_cache)
  • Apache HTTP Server (apr_pool)
  • Game engines for particle systems
  • Database systems for tuple storage

Benchmarking Link to heading

Here’s a simple benchmark comparing slab allocator vs. system malloc:

#include <chrono>
#include <iostream>

void benchmark() {
    const int iterations = 1000000;
    
    // System allocator
    auto start = std::chrono::high_resolution_clock::now();
    std::vector<void*> ptrs;
    for (int i = 0; i < iterations; ++i) {
        ptrs.push_back(::operator new(64));
    }
    for (void* ptr : ptrs) {
        ::operator delete(ptr);
    }
    auto end = std::chrono::high_resolution_clock::now();
    auto system_time = std::chrono::duration_cast<std::chrono::microseconds>(end - start);

    // Slab allocator
    SlabAllocator slab(64);
    start = std::chrono::high_resolution_clock::now();
    ptrs.clear();
    for (int i = 0; i < iterations; ++i) {
        ptrs.push_back(slab.allocate());
    }
    for (void* ptr : ptrs) {
        slab.deallocate(ptr);
    }
    end = std::chrono::high_resolution_clock::now();
    auto slab_time = std::chrono::duration_cast<std::chrono::microseconds>(end - start);

    std::cout << "System allocator: " << system_time.count() << " μs\n";
    std::cout << "Slab allocator: " << slab_time.count() << " μs\n";
    std::cout << "Speedup: " << (double)system_time.count() / slab_time.count() << "x\n";
}

Conclusion Link to heading

Slab allocators are an excellent choice for high-performance systems that allocate many objects of the same size. While they require more upfront design and implementation effort, the performance benefits can be substantial.
